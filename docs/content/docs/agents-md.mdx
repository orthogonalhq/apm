## Overview

APM installs skills to `.skills/@scope/name/SKILL.md` and tracks metadata in `apm-lock.json`. But agents need to know which skills to load. APM defines a reference convention that works across any agent configuration file.

## The `@skill:` convention

Reference installed skills in any markdown file your agent reads:

```markdown
@skill:anthropics/code-review
@skill:cloudflare/code-review
```

The agent sees `@skill:scope/name` and knows to load `.skills/scope/name/SKILL.md`.

This works in any agent configuration file:

| Agent | Config file |
|-------|-------------|
| Claude Code | `CLAUDE.md` |
| Cursor | `.cursorrules` |
| GitHub Copilot | `.github/copilot-instructions.md` |
| Gemini CLI | `GEMINI.md` |
| Any agent | Any markdown the agent reads |

### Example: Claude Code

```markdown
# CLAUDE.md

This is a Next.js e-commerce app. The API is in /app/api.

## Skills

@skill:anthropics/code-review
@skill:cloudflare/code-review
```

### Example: Cursor

```markdown
# .cursorrules

Always use TypeScript strict mode.
Prefer server components.

@skill:anthropics/code-review
```

## Why this approach

**Explicit over implicit.** The user decides which skills each agent sees. A skill installed via APM isn't automatically injected into your agent — you choose where to reference it.

**Portable.** The `@skill:` syntax is plain text. It works in any file, any agent, any editor. No special tooling required to read it.

**Composable.** Different agents can load different skills. Your `CLAUDE.md` might reference different skills than your `.cursorrules`.

**Transparent.** Skills are system prompt instructions. Users should always see exactly what instructions their agent is operating under. No hidden injection.

## apm-lock.json as the index

The lockfile is the single source of truth for installed skills. Since v2, it includes metadata for each package:

```json
{
  "lockfileVersion": 2,
  "packages": {
    "@anthropics/code-review": {
      "sourceRepo": "anthropics/knowledge-work-plugins",
      "sourcePath": "engineering/skills/code-review/SKILL.md",
      "sourceRef": "main",
      "commitSha": "abc123",
      "description": "Review code changes for security, performance, and correctness.",
      "kind": "skill",
      "tags": ["code-review", "security"]
    }
  }
}
```

| Field | Description |
|-------|-------------|
| `sourceRepo` | GitHub repository containing the skill |
| `sourcePath` | Path to SKILL.md within the repo |
| `sourceRef` | Git ref (branch/tag) |
| `commitSha` | Pinned commit (if available) |
| `description` | One-line description from SKILL.md frontmatter |
| `kind` | One of: `skill`, `composite-skill`, `workflow`, `app` |
| `tags` | Tags from SKILL.md frontmatter |

The lockfile provides everything needed for tooling, editors, and agents to understand what skills are available without scanning the `.skills/` directory.

## Agent adoption

Today, no agents natively resolve `@skill:` references. Until they do:

1. **Copy the skill content** — Read the SKILL.md and paste relevant sections into your agent config
2. **Use `@import` if supported** — Some agents (like Claude Code) support importing files: `@import .skills/anthropics/code-review/SKILL.md`
3. **Reference by path** — Point the agent at the file: "Follow the instructions in `.skills/anthropics/code-review/SKILL.md`"

As APM and the agentskills.io ecosystem grow, we expect agents to adopt `@skill:` natively — making skill discovery automatic.

## Version control

| File | Commit? | Reason |
|------|---------|--------|
| `apm-lock.json` | Yes | Reproducible installs and skill index for the team |
| `.skills/` | Optional | Some teams commit skills, others regenerate via `apm install` |

## Project structure

```
your-project/
├── CLAUDE.md              # References @skill:scope/name
├── apm-lock.json          # Lockfile with skill metadata
├── .skills/
│   ├── anthropics/
│   │   └── code-review/
│   │       └── SKILL.md
│   └── cloudflare/
│       └── code-review/
│           └── SKILL.md
└── ... your project files
```
